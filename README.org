#+title: clime
#+options: toc:nil num:nil

Build real CLI tools in pure Emacs Lisp.  Declare commands, options,
and args in a single ~clime-app~ form --- clime handles parsing,
~--help~ generation, error messages, and dispatch.

* Getting Started

Write an Elisp file using the clime DSL:

#+begin_src emacs-lisp
;;; greet.el --- A greeting CLI  -*- lexical-binding: t; -*-

;;; Code:

(require 'clime)

(clime-app greet
  :version "1.0.0"
  :help "A friendly greeter."

  (clime-option shout ("--shout" "-s") :flag t
    :help "SHOUT THE GREETING")

  (clime-command hello
    :help "Say hello to someone"
    (clime-arg name :help "Person to greet")
    (clime-handler (ctx)
      (clime-let ctx (name shout)
        (let ((msg (format "Hello, %s!" name)))
          (if shout (upcase msg) msg)))))

  (clime-command goodbye
    :help "Say goodbye"
    (clime-arg name :help "Person to bid farewell")
    (clime-handler (ctx)
      (clime-let ctx (name)
        (format "Goodbye, %s!" name)))))

(provide 'greet)
;;; greet.el ends here
#+end_src

Run ~quickstart~ to add the entrypoint boilerplate and shebang:

#+begin_src bash
path/to/clime-make.el quickstart greet.el

./greet.el hello world        # => Hello, world!
./greet.el --shout hello world # => HELLO, WORLD!
./greet.el --help
./greet.el hello --help
#+end_src

** How the shebang works

~clime-make.el init~ prepends a two-line polyglot header that's valid as
both shell script and Elisp:

#+begin_src bash
#!/bin/sh
":"; CLIME_ARGV0="$0" exec emacs --batch -Q \
  -L "$(dirname "$0")" \
  --eval "(setq load-file-name \"$0\")" \
  --eval "(with-temp-buffer \
    (insert-file-contents load-file-name) \
    (setq lexical-binding t) \
    (goto-char(point-min)) \
    (condition-case nil \
      (while t(eval(read(current-buffer))t)) \
      (end-of-file nil)))" \
  -- "$@" # clime-sh!:v1 -*- mode: emacs-lisp; lexical-binding: t; -*-
#+end_src

Line 2 is both a shell no-op (~":"~) and the Emacs invocation.  It uses
a read/eval loop that explicitly sets ~lexical-binding~ to ~t~, since
the ~":"~ prefix on line 2 prevents Emacs from recognizing the ~-*-~
cookie during normal file loading.  The cookie is still useful: editors
use it to detect the major mode when opening the file for editing.

The ~# clime-sh!:vN~ tag lets ~init~ detect and update existing
shebangs without clobbering newer formats.  ~CLIME_ARGV0~ preserves
the original script name for usage output.

So ~./myapp.el hello world~ just works from your shell.

* Features

- Declarative DSL --- one form defines your entire CLI
- Subcommands and nested groups (~myapp repo add~)
- Auto-generated ~--help~ and ~--version~ at every level
- Boolean flags, count flags (~-vvv~), repeatable options (~--tag a --tag b~)
- Collapsed short flags (~-abc~ expands to ~-a -b -c~)
- ~--flag=value~ syntax
- Choices validation (~:choices '("json" "csv")~)
- Custom value transforms (~:coerce #'expand-file-name~)
- Type coercion (string, integer, number)
- Default values
- Command aliases (~:aliases (i)~)
- Hidden commands and options
- Environment variables with auto-derived or per-option names
- Stdin via ~-~ sentinel
- JSON output mode (~:json-mode t~)
- Epilog text for extra help content
- Category sections in help output (options, commands, inline groups)
- Rest args (~:nargs :rest~)
- Invocable groups (group with its own handler)
- Inline groups (~:inline t~) --- promote children to parent level
- Command aliases for nested paths (~clime-alias-for start (agents start)~)
- Lazy choices (~:choices~ accepts a function)
- Value conformers (~:conform~) for pass-2 validation and transformation
- Parameter templates (~clime-defopt~, ~clime-defarg~) with ~:from~ inheritance
- Ancestor option propagation --- group options available to all descendants
- Setup hook (~:setup~) with two-pass parse for config loading
- ~clime-params-plist~ --- convert context params to keyword plist
- Single-file distribution via ~bundle~ command

** Options

Boolean flags consume no value.  Use ~:flag t~ (shorthand for ~:nargs 0~):

#+begin_src emacs-lisp
(clime-option force ("--force" "-f") :flag t
  :help "Skip confirmation")
#+end_src

Count flags increment on each occurrence:

#+begin_src emacs-lisp
(clime-option verbose ("-v" "--verbose") :count t
  :help "Increase verbosity")
;; -v → 1, -vv → 2, -vvv → 3
#+end_src

Repeatable options collect values into a list:

#+begin_src emacs-lisp
(clime-option tag ("--tag" "-t") :multiple t
  :help "Add tag")
;; --tag dev --tag ci → ("dev" "ci")
#+end_src

Default values are used when an option isn't provided:

#+begin_src emacs-lisp
(clime-option registry ("--registry" "-r")
  :help "Registry to use"
  :default "default")
#+end_src

** Collapsed short flags

Single-character flags can be collapsed: ~-abc~ expands to ~-a -b -c~.
This works when all flags in the group are boolean.  A trailing
value-taking flag consumes the next argument:

#+begin_src bash
./myapp.el -vvf show 123   # -v -v -f show 123
#+end_src

** ~--flag=value~ and ~--~

Options accept ~--flag=value~ syntax:

#+begin_src bash
./myapp.el show --format=json 123
#+end_src

Use ~--~ to stop option parsing --- everything after is treated as
positional arguments:

#+begin_src bash
./myapp.el show -- --not-a-flag
#+end_src

** Type coercion

Set ~:type~ on an option or arg to convert from the raw string:

#+begin_src emacs-lisp
(clime-option count ("--count" "-n") :type 'integer
  :help "Number of items")

(clime-option rate ("--rate") :type 'number
  :help "Rate as a float")
#+end_src

Supported types: ~string~ (default), ~integer~, ~number~.

** Choices validation

Restrict an option or arg to a fixed set of values:

#+begin_src emacs-lisp
(clime-option format ("--format") :choices '("json" "table" "csv")
  :help "Output format")

(clime-arg action :choices '("start" "stop" "restart")
  :help "Service action")
#+end_src

Invalid values produce a clear error:

#+begin_example
Error: Invalid value "xml" for --format (choose from: json, table, csv)
#+end_example

Choices are validated post-coercion, so they work with ~:type 'integer~:

#+begin_src emacs-lisp
(clime-option level ("--level") :type 'integer :choices '(1 2 3))
#+end_src

~:choices~ also accepts a function for lazy evaluation --- the function
is called at parse time and help time:

#+begin_src emacs-lisp
(clime-option format ("--format")
  :choices (lambda () (compute-available-formats))
  :help "Output format")
#+end_src

** Custom transforms with ~:coerce~

Apply an arbitrary function after type coercion:

#+begin_src emacs-lisp
(clime-option path ("--path") :coerce #'expand-file-name
  :help "File path (expanded)")

(clime-option env ("--env") :choices '("dev" "prod") :coerce #'upcase
  :help "Deploy target")
;; --env dev → "DEV"
#+end_src

When combined with ~:choices~, validation runs first, then ~:coerce~.

** Value conformers with ~:conform~

~:conform~ runs in pass 2 after env vars are applied but before
defaults.  Unlike ~:coerce~ (which runs immediately during parsing),
~:conform~ validates values from both CLI and env var sources.
Defaults skip conforming --- they're developer-authored and should
already be valid:

#+begin_src emacs-lisp
(clime-option id ("--id")
  :conform (lambda (val)
             (unless (string-match-p "^[a-z0-9]\\{7\\}$" val)
               (error "Invalid ID format: %s" val))
             (upcase val))
  :help "Resource ID (7-char alphanumeric)")
#+end_src

A conformer receives the value and must either return the conformed
value or signal an error.  This is useful for:

- Domain validation of external input (CLI args and env vars)
- Normalizing values before they reach the handler
- Cross-cutting checks shared via parameter templates (~:from~)

The pipeline order is: type coercion → ~:coerce~ → choices → env vars → ~:conform~ → defaults.

** Environment variables

Set ~:env-prefix~ on the app to auto-derive env var names from options:

#+begin_src emacs-lisp
(clime-app greet
  :version "1.0.0"
  :help "A friendly greeter."
  :env-prefix "GREET"
  ...)
#+end_src

Now ~--shout~ can also be set via ~GREET_SHOUT=1~.

For individual options, override the env var name with ~:env~:

#+begin_src emacs-lisp
(clime-option token ("--token") :env "API_TOKEN"
  :help "Auth token")
#+end_src

Boolean flags accept ~1~/~true~/~yes~ and ~0~/~false~/~no~.  Multiple
options split on comma: ~MYAPP_TAGS=dev,ci~ becomes ~("dev" "ci")~.

** Aliases

Commands can have aliases:

#+begin_src emacs-lisp
(clime-command install
  :help "Install a package"
  :aliases (i)
  ...)
#+end_src

Now both ~myapp install foo~ and ~myapp i foo~ work.  Aliases are
symbols or strings.

** Command aliases for nested paths

Use ~clime-alias-for~ to expose a nested command at a higher level
without duplicating its definition:

#+begin_src emacs-lisp
(clime-group agents
  :help "Manage agents"
  (clime-command start
    :help "Start an agent"
    (clime-arg name :help "Agent name")
    (clime-option prompt ("--prompt") :help "Initial prompt")
    (clime-handler (ctx) (do-start ctx))))

(clime-group shortcuts :inline t :category "Shortcuts"
  (clime-alias-for start (agents start)
    :help "Start an agent (shortcut)"))
#+end_src

Now ~myapp start foo~ works identically to ~myapp agents start foo~.
The alias copies args, options, and handler from the target at init
time.  You can override ~:help~, ~:aliases~, ~:hidden~, and
~:category~ on the alias.  When ~:help~ is omitted, it inherits from
the target.

** Hidden commands and options

Mark a command or option as ~:hidden t~ to omit it from ~--help~ while
keeping it callable:

#+begin_src emacs-lisp
(clime-command debug
  :help "Dump internal state"
  :hidden t
  (clime-handler (ctx) ...))

(clime-option trace ("--internal-trace") :flag t
  :help "Enable tracing" :hidden t)
#+end_src

** Nested groups

Organize related commands under a group:

#+begin_src emacs-lisp
(clime-group repo
  :help "Manage repositories"

  (clime-command add
    :help "Add a repository"
    (clime-arg name :help "Repository name")
    (clime-arg url :help "Repository URL")
    (clime-handler (ctx)
      (clime-let ctx (name url)
        (format "Added %s → %s" name url))))

  (clime-command remove
    :help "Remove a repository"
    :aliases (rm)
    (clime-arg name :help "Repository name")
    (clime-handler (ctx)
      (clime-let ctx (name) (format "Removed %s" name)))))
#+end_src

** Inline groups

Mark a group with ~:inline t~ to promote its children to the parent
level.  Both the group prefix and the short form work:

#+begin_src emacs-lisp
(clime-group repo
  :inline t
  :help "Manage repositories"

  (clime-command add
    :help "Add a repository"
    (clime-arg name :help "Repository name")
    (clime-handler (ctx)
      (clime-let ctx (name) (format "Added %s" name))))

  (clime-command remove
    :help "Remove a repository"
    (clime-arg name :help "Repository name")
    (clime-handler (ctx)
      (clime-let ctx (name) (format "Removed %s" name)))))
#+end_src

#+begin_src bash
./myapp.el add foo       # works (promoted)
./myapp.el repo add foo  # also works (direct)
#+end_src

In ~--help~, inline group children appear at the parent level.  The
group name itself is hidden from the command listing.  If a parent has
its own child with the same name, the parent's child takes priority.

** Invocable groups

A group can have its own ~:handler~ that runs when no subcommand is
given:

#+begin_src emacs-lisp
(clime-group config
  :help "View or modify configuration"

  (clime-handler (_ctx)
    "Configuration:\n  registry = default\n  timeout = 30s")

  (clime-command set
    :help "Set a config value"
    (clime-arg key :help "Config key")
    (clime-arg value :help "New value")
    (clime-handler (ctx)
      (clime-let ctx (key value)
        (format "Set %s = %s" key value)))))
#+end_src

~myapp config~ shows the overview; ~myapp config set key val~ sets a value.

** Rest args

Collect all remaining positional arguments into a list:

#+begin_src emacs-lisp
(clime-command run
  :help "Run a script"
  (clime-arg script :help "Script name")
  (clime-arg args :nargs :rest :required nil
    :help "Arguments passed to the script")
  (clime-handler (ctx)
    (clime-let ctx (script args)
      (format "Running %s with %s" script args))))
#+end_src

** Epilog

Add free-form text after the auto-generated help with ~:epilog~:

#+begin_src emacs-lisp
(clime-app myapp
  :version "1.0"
  :help "My application."
  :epilog "Examples:
  myapp install foo --tag dev
  myapp config set timeout 60"
  ...)
#+end_src

Commands and groups support ~:epilog~ too.

** ~:doc~ alias

~:doc~ is an alias for ~:help~ in the DSL:

#+begin_src emacs-lisp
(clime-command show
  :doc "Show a resource"
  ...)
#+end_src

** JSON output mode

Set ~:json-mode t~ on the app to auto-inject a ~--json~ flag.
When ~--json~ is set, handler return values are encoded as JSON:

#+begin_src emacs-lisp
(clime-app myapp
  :version "1.0"
  :json-mode t
  ...)
#+end_src

#+begin_src bash
./myapp.el show --json 123
#+end_src

** Categories in help

Use ~:category~ on options, commands, or inline groups to organize
~--help~ into named sections.  Items sharing a category appear under
one heading; uncategorized items fall back to "Options:" or "Commands:".

Options with ~:category~:

#+begin_src emacs-lisp
(clime-option author ("--author" "-a")
  :help "Filter by author"
  :category "Filter")

(clime-option since ("--since")
  :help "Only after DATE"
  :category "Filter")
#+end_src

These appear under a "Filter:" section instead of "Options:".

Commands with ~:category~:

#+begin_src emacs-lisp
(clime-command search
  :help "Search items" :category "Filter")
(clime-command list
  :help "List items" :category "Filter")
(clime-command show
  :help "Show details")   ;; uncategorized → "Commands:"
#+end_src

Options and commands with the same ~:category~ share a section.

Inline groups with ~:category~ apply it to all their children and
options:

#+begin_src emacs-lisp
(clime-group admin
  :inline t :category "Admin"
  :help "Administrative commands"

  (clime-option admin-token ("--admin-token")
    :help "Admin API token")

  (clime-command status
    :help "Show system status"))
#+end_src

This renders as:

#+begin_example
Admin:
  Administrative commands

  --admin-token VALUE    Admin API token
  status                 Show system status
#+end_example

Nested inline groups compose category paths --- an inner group with
~:category "Config"~ inside an outer ~:category "Admin"~ produces
"Admin / Config:" in help.

** Ancestor option propagation

Options declared at any level are automatically available to all
descendants.  A ~--verbose~ flag on the app or on a group can be used
from any child command:

#+begin_src emacs-lisp
(clime-app myapp
  :version "1.0"

  (clime-option verbose ("-v" "--verbose") :count t
    :help "Increase verbosity")

  (clime-group repo
    :help "Manage repositories"
    (clime-option registry ("--registry") :help "Registry URL")

    (clime-command add
      :help "Add a repo"
      (clime-arg name :help "Repo name")
      ;; Handler can access verbose (from app) and registry (from group)
      (clime-handler (ctx)
        (clime-let ctx (name verbose registry)
          (format "Adding %s (verbose=%s, registry=%s)" name verbose registry))))))
#+end_src

In ~--help~, inherited options appear in a separate "Global Options:"
section.

** Setup hook

The ~:setup~ hook runs after pass-1 parsing (all values extracted from
argv) but before pass-2 finalization (dynamic ~:choices~ validation,
env vars, defaults).  This lets apps load configuration that influences
dynamic option values:

#+begin_src emacs-lisp
(clime-app myapp
  :version "1.0"
  :env-prefix "MYAPP"
  :setup (lambda (app result)
           (let ((dir (plist-get (clime-parse-result-params result) 'config-dir)))
             (my-load-config (or dir "~/.myapp"))))

  (clime-option config-dir ("--config-dir")
    :help "Config directory")

  (clime-option level ("--level")
    :help "Log level"
    :choices #'my-configured-levels)  ; resolved after setup

  (clime-command run
    :help "Run the app"
    (clime-handler (ctx) "running")))
#+end_src

Static ~:choices~ (literal lists) are validated immediately in pass 1.
Dynamic ~:choices~ (functions) are deferred to pass 2 after the setup
hook has run.

** Parameter templates

Define reusable option or arg specs with ~clime-defopt~ and
~clime-defarg~, then reference them with ~:from~:

#+begin_src emacs-lisp
;; Define a template once
(clime-defopt verified-id
  :type 'string
  :conform #'my-ensure-id
  :help "A verified resource ID")

;; Reuse across commands — explicit values override the template
(clime-command copy
  :help "Copy a resource"
  (clime-option src ("--src") :from verified-id :multiple t)
  (clime-option dst ("--dst") :from verified-id)
  (clime-handler (ctx) ...))

(clime-command move
  :help "Move a resource"
  (clime-option src ("--src") :from verified-id)
  (clime-option dst ("--dst") :from verified-id)
  (clime-handler (ctx) ...))
#+end_src

~clime-defopt~ expands to a ~defvar~ with a ~clime--opt-~ prefix
(e.g. ~clime--opt-verified-id~).  The ~:from~ keyword handles the
prefixing transparently --- you write the bare name in both places.

Templates cannot contain ~:name~ or ~:flags~ (those are per-instance).
All DSL shorthands (~:flag t~, ~:separator~) work in templates.

~clime-defarg~ works the same way for positional args:

#+begin_src emacs-lisp
(clime-defarg file-path
  :type 'string
  :coerce #'expand-file-name
  :help "Path to a file")

(clime-command cat
  :help "Print file contents"
  (clime-arg path :from file-path)
  (clime-handler (ctx) ...))
#+end_src

** ~clime-params-plist~

Convert context params to a keyword plist, useful for passing parsed
values to other functions:

#+begin_src emacs-lisp
(clime-handler (ctx)
  ;; All params as keyword plist
  (my-function (clime-params-plist ctx))
  ;; => (:verbose 3 :package "foo" :force t)

  ;; Selected params only (nil values omitted)
  (my-other-function (clime-params-plist ctx 'package 'force)))
#+end_src

** Stdin

Use ~-~ as an argument value to read from stdin:

#+begin_src bash
echo "World" | ./greet.el hello -
#+end_src

** Help

~--help~ and ~-h~ are recognized at every level.  Both orderings work:

#+begin_src bash
./myapp.el install --help   # help for install
./myapp.el --help install   # same thing
./myapp.el --help           # app-level help
#+end_src

Usage errors include a hint:

#+begin_example
Error: Missing required argument <name>
Try 'myapp install --help' for more information.
#+end_example

Multiline ~:help~ strings are truncated to the first line in command
listings, and shown in full in detailed command help.

* DSL Reference

| Macro            | Purpose                            | Key arguments                                                             |
|------------------+------------------------------------+---------------------------------------------------------------------------|
| ~clime-app~      | Define a CLI application           | ~:version~, ~:help~, ~:env-prefix~, ~:json-mode~, ~:epilog~, ~:setup~   |
| ~clime-command~  | Define a subcommand                | ~:help~, ~:aliases~, ~:hidden~, ~:epilog~, ~:category~                   |
| ~clime-group~    | Nest related commands              | ~:help~, ~:aliases~, ~:hidden~, ~:inline~, ~:epilog~, ~:category~        |
| ~clime-option~   | Declare a named option             | ~:flag~, ~:count~, ~:multiple~, ~:type~, ~:choices~, ~:coerce~, ~:conform~, ~:env~, ~:category~, ~:from~ |
| ~clime-arg~      | Declare a positional argument      | ~:nargs~, ~:type~, ~:choices~, ~:coerce~, ~:conform~, ~:default~, ~:help~, ~:from~ |
| ~clime-defopt~   | Define a reusable option template  | Any option slot except ~:name~, ~:flags~                                 |
| ~clime-defarg~   | Define a reusable arg template     | Any arg slot except ~:name~                                              |
| ~clime-alias-for~ | Alias for a nested command        | ~:help~, ~:aliases~, ~:hidden~, ~:category~                             |
| ~clime-handler~  | Attach a handler to a command      | ~(ctx)~ --- receives the parse context                                   |
| ~clime-let~      | Destructure context in a handler   | ~ctx (var1 var2 ...)~ --- binds option/arg values                        |

~:doc~ is an alias for ~:help~ on commands, groups, and apps.

~:flag t~ is shorthand for ~:nargs 0~ (boolean flag, DSL only).

See =examples/pkm.el= for a comprehensive demo covering all major features.

* Bundling & Distribution

~clime-make.el~ provides four commands:

| Command        | Purpose                                          |
|----------------+--------------------------------------------------|
| ~init~         | Add polyglot shebang to make a script executable |
| ~scaffold~     | Insert ~;;; Entrypoint:~ boilerplate              |
| ~quickstart~   | ~scaffold~ + ~init~ in one shot (auto env vars)  |
| ~bundle~       | Combine multiple source files into one            |

** Making a script executable (~init~)

For a single-file app that lives alongside clime:

#+begin_src bash
./clime-make.el init myapp.el
./myapp.el --help
#+end_src

~init~ prepends a polyglot shebang, sets the executable bit, and adds
~-L~ flags so Emacs can find clime at runtime.

Common options:

#+begin_src bash
# Script in a subdirectory that needs the parent on the load path
./clime-make.el init -R .. examples/myapp.el

# Script's own dir on the load path (for local requires)
./clime-make.el init --self-dir myapp.el

# Bundled/vendored app that doesn't need the clime load path
./clime-make.el init --standalone --self-dir dist/myapp.el

# Set env vars in the shebang
./clime-make.el init --env CLIME_MAIN_APP=myapp dist/myapp.el
#+end_src

Re-running ~init~ on a file that already has a clime shebang updates
it in place.  ~init~ refuses to overwrite a non-clime shebang (use
~--force~) or to downgrade a newer shebang format version.

** Inserting entrypoint boilerplate (~scaffold~)

~scaffold~ detects the ~clime-app~ symbol in a file and inserts an
~;;; Entrypoint:~ section with the ~clime-main-script-p~ guard:

#+begin_src bash
./clime-make.el scaffold myapp.el
#+end_src

Before:

#+begin_src emacs-lisp
(clime-app myapp :version "1.0" ...)
(provide 'myapp)
;;; myapp.el ends here
#+end_src

After:

#+begin_src emacs-lisp
(clime-app myapp :version "1.0" ...)
(provide 'myapp)
;;; Entrypoint:
(when (clime-main-script-p 'myapp)
  (clime-run-batch myapp))
;;; myapp.el ends here
#+end_src

The guard symbol comes from ~(provide 'FEATURE)~ if present, falling
back to the ~clime-app~ symbol.  If ~;;; Entrypoint:~ already exists,
~scaffold~ skips without error.  If there is no ~;;; ... ends here~
marker, the entrypoint is appended at the end.

** One-command quickstart (~quickstart~)

~quickstart~ composes ~scaffold~ + ~init~ and auto-detects
~CLIME_MAIN_APP~:

#+begin_src bash
./clime-make.el quickstart myapp.el
./myapp.el --help
#+end_src

This is the fastest way to go from a source file to a runnable script.
~quickstart~ accepts all of ~init~'s flags (~--self-dir~, ~-R~, ~-L~,
~--standalone~, ~--force~, ~-e~):

#+begin_src bash
./clime-make.el quickstart --self-dir -e MY_VAR=hello myapp.el
#+end_src

An explicit ~-e CLIME_MAIN_APP=custom~ overrides the auto-detected
value.

** Building a single-file distribution (~bundle~)

For multi-file projects, ~bundle~ concatenates source files into one:

#+begin_src bash
./clime-make.el bundle -o dist/myapp.el \
    --provide myapp \
    src/myapp-core.el src/myapp-commands.el src/myapp.el
#+end_src

The output provides the feature (~'myapp~) and can be used as a
library via ~(require 'myapp)~.

*** Adding an entrypoint (~--main~)

To make the bundle also work as a standalone CLI, pass ~--main~:

#+begin_src bash
./clime-make.el bundle -o dist/myapp.el \
    --provide myapp --main myapp-main.el \
    src/myapp-core.el src/myapp-commands.el src/myapp.el

./clime-make.el init --standalone --self-dir \
    --env CLIME_MAIN_APP=myapp dist/myapp.el
#+end_src

The main file is a short script --- it requires the app and runs it:

#+begin_src emacs-lisp
;;; myapp-main.el --- Entrypoint for myapp  -*- lexical-binding: t; -*-
;;; Code:
(require 'myapp)
(clime-run-batch myapp)
;;; myapp-main.el ends here
#+end_src

Its code is wrapped in a ~clime-main-script-p~ guard inside the
bundle, so it only executes when ~CLIME_MAIN_APP~ matches the feature
name --- not when loaded as a library.  The ~--main~ file must /not/
contain an =;;; Entrypoint:= marker.

*** Entrypoint markers in source files

Source files can contain an =;;; Entrypoint:= section that separates
library code from standalone entrypoint code:

#+begin_src emacs-lisp
;;; myapp.el --- My app  -*- lexical-binding: t; -*-
;;; Code:
(require 'clime)
(clime-app myapp :version "1.0" ...)
(provide 'myapp)
;;; Entrypoint:
(clime-run-batch myapp)
;;; myapp.el ends here
#+end_src

The bundler extracts only the library section (=;;; Code:= to
=;;; Entrypoint:=), stripping the entrypoint automatically.  This lets
each source file run standalone during development while bundling
cleanly.  Files without the marker are extracted up to
=;;; ... ends here= as usual.

If ~(clime-run-batch ...)~ appears in a source file's library section
(above the marker or without one), ~bundle~ signals an error --- move
it below =;;; Entrypoint:=.

*** Typical project layout

#+begin_example
myapp/
  vendor/clime/          # clime source (git clone or submodule)
  src/
    myapp-core.el        # library code
    myapp-commands.el    # commands
    myapp.el             # app definition + (provide 'myapp)
  myapp-main.el          # standalone entrypoint
  dist/
    myapp.el             # bundled output (single file)
  Makefile
#+end_example

A minimal Makefile:

#+begin_src makefile
CLIME := vendor/clime/clime-make.el
SRCS  := src/myapp-core.el src/myapp-commands.el src/myapp.el

dist/myapp.el: $(SRCS) myapp-main.el
	$(CLIME) bundle -o $@ --provide myapp --main myapp-main.el $(SRCS)
	$(CLIME) init --standalone --self-dir --env CLIME_MAIN_APP=myapp $@
#+end_src

* Installation

** Git clone / submodule

#+begin_src bash
# Clone directly
git clone https://github.com/cosmicz/clime vendor/clime

# Or as a submodule
git submodule add https://github.com/cosmicz/clime vendor/clime
#+end_src

Then set up your app file:

#+begin_src bash
vendor/clime/clime-make.el quickstart myapp.el
#+end_src

This inserts the entrypoint boilerplate, prepends a polyglot shebang,
and makes the file executable.

** package-vc (Emacs 29+)

#+begin_src emacs-lisp
(package-vc-install "https://github.com/cosmicz/clime")
#+end_src

** Vendoring (single file)

Build a single-file bundle and copy it into your project:

#+begin_src bash
cd clime/
make dist
cp dist/clime.el /path/to/myapp/
#+end_src

Your app just needs ~(require 'clime)~ and ~-L~ pointing to the
directory containing the bundled file.

* Prior Art

clime draws inspiration from several projects:

  subcommands, options, and completions
